## Technical Details

### Files Modified
- [projects/mmv-tarots/app/page.tsx](projects/mmv-tarots/app/page.tsx)

### Key Code Changes
- Changed `min-h-[100dvh]` to `h-[calc(100dvh-64px)]` (mobile) and `h-[calc(100dvh-80px)]` (desktop) to account for the header.
- Removed `pb-32` from the main container.
- Reduced `pt-20` to `pt-10` on the `h1` for mobile.
- Reduced `pt-12` to `pt-6` on the sticky input container.
- Added `pb-40` to `main-content` to ensure content isn't obscured by the fixed input box.

### Architecture Decisions
- **Viewport-based Height**: Used `calc` with `100dvh` to ensure the container fits exactly within the visible area minus the header, preventing the browser from adding a scrollbar by default.

## AI Diary (REQUIRED - min 150 words)
I started this session with a clear goal: make the home page more accessible by ensuring the core interaction elements are visible immediately. I assumed that the scrolling issue was just a matter of too much content, but I learned when I looked at the code that it was actually a "forced" scroll caused by a combination of `min-h-[100dvh]` and a large `pb-32` padding. I was confused about why the input box felt so far down until I realized the `h1` had a massive top padding on mobile that pushed everything else out of view. I expected a simple padding adjustment would fix it, but I got a more robust solution by calculating the exact height needed to fit the viewport because I wanted to ensure that even on smaller devices, the "Sacred Space" of the app remains sacred and uncluttered. This session felt like a surgical strike—identifying the exact CSS classes that were causing friction and replacing them with more intentional layout logic. It's a reminder that "Above the Fold" isn't just a marketing term; it's about respecting the user's immediate attention.
